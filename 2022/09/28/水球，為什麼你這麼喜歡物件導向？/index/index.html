水球，你為什麼這麼喜歡物件導向？

發表於2022-09-28|更新於2022-10-04|軟工札記
|閱讀量:

水球，你為什麼這麼喜歡物件導向？

我認識物件導向，到對其著迷的這段過程還挺純粹且幼稚的。有許多學生們學習物件導向只是為了熟悉某個 OOP 語言，但其實這樣子對於物件導向的理解十分狹隘，而且你可能無法感受到龐大的樂趣。因此我想將我對於物件導向的初衷分享給各位。

大一時，我也是在 C++ 中寫著 Procedure-oriented 的程式碼，下著一道一道的指令。當時雖然就有老師在宣傳業界的「物件導向」、「模組化」等實踐原則，但這些詞彙從老師嘴裡說出來聽起來十分抽象，完全不可能得到學生的注意。

「物件導向」？Who cares 「物件」？我只知道期末專案要考的是「萬年曆」的演算法。一道道指令的淹沒下，只是為了在小黑視窗中印出一個超陽春的月曆。當時我根本想不到 programming paradigm 能有其他種變化。

直到我在大二時看到的這一行程式碼：panel→buttons[3]→click() ，是在 C++ Builder 寫資料結構作業時看到的。當時的我認為這一行程式碼很有趣，覺得這是什麼鬼？怎麼這麼多箭頭，但是好像我又總能很順地理解過去。

「面板中的・按鈕中・索引 3 的按鈕・將它按下去」——這是物件導向程式最基本的特徵，能夠對物件做一連串的存取。如果只是這一串存取式，那麼其實並沒有多少魅力。但我腦海中卻立刻浮現了其他描述：「英雄身上的・背包中的・第七個欄位・使用這個欄位的道具」⋯⋯ hero.bag.slots[7].use() ——我腦袋的直接反應竟是將以前沈迷電玩遊戲時，對於遊戲世界結構的認知，都吻合上了類似的邏輯，然後自然而然地衍生出許多類似的 OOP 程式碼。

自這一秒之後，我繼續翻翻找找相關的程式碼，上網查查那個箭頭的意思（查得一塌糊塗）。慢慢得，我才開始想到教授所說的「物件導向」，會不會指的就是這個意思呢？若我所撰寫的程式，看起來像是在組織一群活生生物件們，那這種圍繞在物件彼此關係和互動的程式撰寫風格，是不是就是「物件導向」程式呢？

從此之後，我就開始不斷摸索怎麼樣創造出一個物件，我想要宣告一個「水球」物件，水球有 50 滴生命值和 60 滴魔力。無奈，我在 C++ 中怎麼試都試不出這個結果，後來才發現，我得先知道水球物件是什麼「類別」。

因此可以說呀，我對於物件導向的學習歷程比較特殊，全部源自於我對遊戲世界的各種物件模擬的憧憬，而不是源自於 Java 對 OOP 的規範、SOLID 原則、MVC 分層、軟體維護性、擴充性、可讀性等等、等等，毫無熱情的字眼。

我喜歡物件導向，而且不斷精進物件導向造詣的初衷就是因為我是一位「模擬控」罷了，哈哈。所以你問我，為什麼我這麼喜歡物件導向，甚至還願意花這麼多時間去精進物件導向的軟體設計能力？

這個問題的答案很簡單，你可以去問問那些可以玩 Minecraft 玩一整天的小朋友們，問問他們為什麼要蓋一棟可以自動化的高科技建築，其中每個物件彼此之間的互動都被巧妙地設計和安裝，為的是什麼呢？

為的就是看到 These Objects Work like a Charm 啊！

至於設計模式就是另外一則故事了，下集待續。
